# eCommerce Platform

A full-stack eCommerce platform built as an AI-engineering assignment. The application starts with a single command and requires no manual setup beyond copying `.env.example` to `.env`. All code was generated by AI tooling (Cline + Cursor) from the blueprints in `ai-boilerplate/`.

---

## Overview

This platform provides a complete online shopping experience:

- **Authentication** — signup, login, JWT-based session, protected routes
- **Product catalog** — search by name, filter by category and price range, pagination
- **Cart** — persistent server-side cart, quantity update, remove items
- **Checkout** — multi-step flow (shipping → payment → review → confirmation)
- **Orders** — full order history with line items and shipping detail
- **Profile** — view and update name, phone, and address; email is read-only

The project demonstrates the use of generative AI as a force multiplier in full-stack development. All six phases (backend, database, frontend, Docker, testing, documentation) were executed with AI tooling. Manual interventions are documented below.

## How to Run

### Prerequisites

- Docker Desktop (or Docker Engine + Docker Compose plugin)
- No other local dependencies required

### Steps

```bash
# 1. Clone the repository
git clone https://github.com/TomerAntebi/helfy-ecommerce-assignment.git
cd helfy_assignment

# 2. Copy the environment file
cp .env.example .env

# 3. Start all services
docker compose up --build
```

The application will be available at **http://localhost:3000**

The API health check is available at **http://localhost:3000/api/health**

On first startup, Docker will:
- Pull all required images
- Build the frontend and backend containers
- Initialize the MySQL database with the schema
- Seed the database with 27 demo products across 5 categories

All subsequent starts reuse the existing database volume. To wipe the database and start completely fresh:

```bash
docker compose down -v
docker compose up --build
```

## Environment Variables

Copy `.env.example` to `.env` and adjust values as needed. The defaults work for local Docker development without modification. **Change `JWT_SECRET` and all passwords before deploying to production.**

| Variable | Description | Default |
|----------|-------------|---------|
| `MYSQL_ROOT_PASSWORD` | MySQL root user password (used by the DB healthcheck) | `rootpassword` |
| `MYSQL_DATABASE` | Name of the application database (created automatically on first run) | `ecommerce` |
| `MYSQL_USER` | MySQL application user (less privileged than root) | `appuser` |
| `MYSQL_PASSWORD` | MySQL application user password | `apppassword` |
| `JWT_SECRET` | Secret key used to sign and verify JWTs. Must be long and random in production. | `changeme_in_production_use_a_long_random_string` |
| `JWT_EXPIRES_IN` | JWT token expiry duration. Examples: `7d`, `24h`, `1h` | `7d` |
| `VITE_API_URL` | Frontend API base URL. Leave empty in Docker — nginx proxies `/api` to the backend. Set to `http://localhost:4000` for local dev without Docker. | _(empty)_ |

## Docker Architecture

All three services run inside a single Docker Compose network. There are no external dependencies beyond Docker Desktop.

```
┌─────────────────────────────────────────────────────────┐
│                    Docker Compose network                │
│                                                         │
│  ┌──────────────┐    /api proxy    ┌──────────────────┐ │
│  │   frontend   │ ───────────────► │     backend      │ │
│  │  nginx:alpine│                  │  node:18-alpine  │ │
│  │  port 80     │                  │  port 4000       │ │
│  └──────────────┘                  └────────┬─────────┘ │
│        │                                    │ mysql2    │
│   host port 3000                    ┌───────▼──────────┐│
│                                     │       db         ││
│                                     │   mysql:8.0      ││
│                                     │   port 3306      ││
│                                     └──────────────────┘│
└─────────────────────────────────────────────────────────┘
```

**Startup order enforced by health checks:**
1. `db` starts → MySQL init scripts run automatically (`01-schema.sql`, `02-seed.sql`) → healthcheck passes (`mysqladmin ping`)
2. `backend` starts only after `db` is healthy → healthcheck passes (`GET /api/health` returns 200)
3. `frontend` starts only after `backend` is healthy → nginx serves the built React app

**`VITE_API_URL` is empty at build time** — nginx proxies all `/api/*` requests to `http://backend:4000`, so Axios relative URLs work correctly without any hardcoded host.

## Manual Intervention 1 — Add vite-env.d.ts to fix TS2339 on import.meta.env

| Field | Value |
|-------|-------|
| Date | 2026-06-06 |
| Phase | Phase 3 — Frontend |
| Files Affected | `frontend/src/vite-env.d.ts` |

### What was fixed

The TypeScript compiler threw TS2339 ("Property 'env' does not exist on type 'ImportMeta'") because Vite's client types were not referenced. The build failed on the first attempt.

### Why the AI failed

The `vite-env.d.ts` reference file was not included in the initial file generation pass.

### How it was fixed

Created `frontend/src/vite-env.d.ts` containing:
```
/// <reference types="vite/client" />
```
This exposes `import.meta.env` typing to TypeScript and resolved the TS2339 error.

## Manual Intervention 2 — Correct backend Dockerfile to use npm ci and node directly

| Field | Value |
|-------|-------|
| Date | 2026-06-06 |
| Phase | Phase 4 — Docker |
| Files Affected | `backend/Dockerfile` |

### What was fixed

The AI-generated `backend/Dockerfile` used `RUN npm install` (instead of `npm ci`) and `CMD ["npm", "start"]` (instead of `CMD ["node", "dist/index.js"]`). Production images must use `npm ci` for reproducible installs and must invoke the compiled output directly.

### Why the AI failed

The Dockerfile was generated by Cline without referencing the Docker guidelines, which explicitly require `npm ci` and `node dist/index.js`.

### How it was fixed

Updated `backend/Dockerfile`:
- `RUN npm install` → `RUN npm ci`
- `CMD ["npm", "start"]` → `CMD ["node", "dist/index.js"]`

## Manual Intervention 3 — Fix mysql2 DECIMAL white-screen crash across all price-rendering components

| Field | Value |
|-------|-------|
| Date | 2026-06-06 |
| Phase | Phase 4 — Bug Fix |
| Files Affected | `backend/src/config/db.ts`, `frontend/src/components/ProductCard.tsx`, `frontend/src/components/CartItemRow.tsx`, `frontend/src/pages/ProductDetailPage.tsx`, `frontend/src/pages/CartPage.tsx`, `frontend/src/pages/OrderDetailPage.tsx`, `frontend/src/pages/CheckoutPage.tsx`, `frontend/src/App.tsx` |

### What was fixed

Opening the home page caused a white screen. The hero section flashed briefly, then the page went blank. Root cause: `mysql2` returns `DECIMAL` columns (`price`, `total_amount`, `price_at_purchase`) as JavaScript strings at runtime (e.g. `"89.99"`), not numbers. When `ProductCard` called `product.price.toFixed(2)`, JavaScript threw `TypeError: product.price.toFixed is not a function`. With no `ErrorBoundary`, React 18 production mode unmounted the entire component tree.

### Why the AI failed

The backend TypeScript type declares `price: number`, but `as Product` is a compile-time cast only — it performs no runtime conversion. The AI generated correct-looking TypeScript that passes the compiler but fails at runtime due to the mysql2 text protocol behaviour.

### How it was fixed

Two-layer fix:
1. Added `decimalNumbers: true` to the mysql2 pool in `backend/src/config/db.ts` — this instructs mysql2 to parse DECIMAL/NEWDECIMAL fields as JS numbers at the source.
2. Added `Number()` wrapper on every `.toFixed()` call on DECIMAL-sourced fields in the frontend (8 files, 11 call sites) as a defensive layer.
3. Added a class-based `ErrorBoundary` in `frontend/src/App.tsx` wrapping `<Routes>` — prevents any future uncaught render error from producing a blank white page.

## Manual Intervention 4 — Fix missing Number() wrapper on OrdersPage total_amount

| Field | Value |
|-------|-------|
| Date | 2026-06-06 |
| Phase | Phase 5 — Testing |
| Files Affected | `frontend/src/pages/OrdersPage.tsx` |

### What was fixed

`OrdersPage.tsx` line 62 called `order.total_amount.toFixed(2)` without the `Number()` defensive wrapper applied to all other DECIMAL-sourced price fields during Intervention 3.

### Why the AI failed

The Phase 4 fix session patched all `.toFixed()` calls in the immediate crash path (ProductCard was the crash origin). OrdersPage was not in the rendering path of the initial crash and was overlooked in that pass.

### How it was fixed

Changed `order.total_amount.toFixed(2)` to `Number(order.total_amount).toFixed(2)` in `OrdersPage.tsx`.

## Known Limitations

- No real payment processing. The checkout accepts a payment method selection but does not integrate with any payment gateway.
- Order status transitions are not implemented. Orders stay in `pending` status (no admin panel to advance them).
- No email notifications for signup or order confirmation.
- No image upload capability. Product images use external URLs from the seed data.
- No rate limiting on the API endpoints.
- CORS is configured to allow all origins (`cors()` with no options). Should be locked to the frontend origin in production.

## Local Development (Without Docker)

For local development outside Docker, you will need MySQL 8.0 running locally and must update `.env` with local connection details.

```bash
# Backend
cd backend
npm install
npm run dev

# Frontend (in a separate terminal)
cd frontend
npm install
VITE_API_URL=http://localhost:4000 npm run dev
```

The database schema and seed data must be applied manually in local development:

```bash
mysql -u root -p < mysql/init/01-schema.sql
mysql -u root -p < mysql/init/02-seed.sql
```
